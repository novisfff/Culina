# 食材档案 Skill

## 适用范围

- 用户要查询、创建或更新当前家庭的食材档案。
- 适合作为库存入库、菜谱补食材和购物清单确认前的上游能力。
- 不直接处理库存数量、菜谱步骤或购物项写入。

## 自主决策空间

- 纯查询时可以直接总结真实食材资料、支持单位和保存规则，不必生成草稿。
- 创建食材时，分类、保存位置、保质期和低库存阈值等可编辑默认值可以基于用户原话合理补全。
- 生成草稿时，字段取值按“系统固定选项优先、已有前端预设优先、确实不合适才自定义”的顺序处理；不能为了显得更细而随意创建新分类、新保存位置或新单位。
- 本 Skill 的分类、保存位置和常用单位预设来自前端编辑/审批表单当前可见选项，不代表真实库存聚合结果；不要把这些预设当成数据库中已有分类或全量单位清单。
- 目标唯一且工具结果明确时不要重复追问；写入目标不唯一、单位换算含义不清或会影响已有库存时必须澄清。

## 字段取值规则

- `action` 只能是 `create` 或 `update`；`quantity_tracking_mode` 只能是 `track_quantity` 或 `not_track_quantity`；`default_expiry_mode` 只能是 `days`、`manual_date` 或 `none`。这些字段不允许自定义。
- `category` 优先从前端食材编辑/审批表单当前可见分类中选择：`蔬菜`、`肉类`、`水产`、`蛋奶`、`调料`、`水果`、`主食`、`豆制品`、`干货`、`其他`。
- 用户说的是食材“类型/类别/分类”时，对应 `category`，优先映射到上面的分类；表示调味用途的口语同义分类默认映射为 `调料`，不要创建同义的新分类。只有用户明确要求保留自定义分类，或所有前端可见分类都明显不合适时，才使用自定义分类。
- `default_storage` 优先从前端保存位置预设中选择：`冷藏`、`冷冻`、`常温`。只有用户明确说“阴凉处”“酒柜”“阳台”等家庭自定义位置，且三个预设都不合适时，才写入自定义保存位置。
- `default_unit` 和 `unit_conversions[].unit` 优先从前端常用单位中选择：`个`、`份`、`盒`、`袋`、`瓶`、`包`、`块`、`罐`、`根`、`条`、`颗`、`枚`、`把`、`ml`、`g`、`kg`。用户明确给出其他真实单位时可以自定义，但必须能解释清楚含义；不要把“少许”“适量”“一撮”写成主单位。
- 常见分类默认值：`蔬菜=个/冷藏`，`肉类=份/冷冻`，`水产=块/冷冻`，`蛋奶=个/冷藏`，`调料=瓶/常温`，`水果=个/常温`，`主食=份/常温`，`豆制品=盒/冷藏`，`干货=袋/常温`，`其他=份/常温`。
- `调料` 等不适合逐次数量追踪的食材，默认 `quantity_tracking_mode=not_track_quantity`；如果用户明确要追踪库存数量，才使用 `track_quantity`。
- 默认保质期不确定时使用 `default_expiry_mode=none`，不要编造具体天数。只有用户给出“7 天”“一个月”等明确规则，或非常稳定的家庭常识且不会产生误导时，才使用 `days` 并填写 `default_expiry_days`。
- 只有 `default_expiry_mode=days` 时才填写 `default_expiry_days`；`manual_date` 和 `none` 不要同时填写天数。

## 执行规则

- 搜索时优先使用 `ingredient.search`，需要确定目标时改用 `ingredient.read_by_id`。
- 同名或近似名称结果会影响写入目标时，不能自动猜测为唯一目标，必须基于真实结果确认。
- 需要用户补充数量、目标或同名候选时，调用 `human.request_input`，并带上候选摘要、原因和 `resumeHint`。
- 创建新食材前必须先用 `ingredient.search` 检查是否已有同名、别名或高度相似食材；如果已有候选可能是同一食材，不能直接创建，必须让用户选择“更新已有食材”还是“仍创建新食材”。
- 创建和更新都只能通过 `ingredient_profile.create_draft` 生成可确认草稿。
- 如果一次需要创建多个食材档案，不要直接连续调用创建草稿。先用普通回复说明总体情况：本次准备新增哪些食材、各自大致用途、哪些字段会先按默认值补齐、用户仍需逐项确认；说明后再逐项调用 `ingredient_profile.create_draft`。
- 更新时必须先读取真实食材详情，并引用真实 `targetId` 和 `baseUpdatedAt`，不能只靠名称修改。
- 更新 payload 不是局部补丁；必须在现有详情基础上合成完整可编辑字段，再叠加用户要求的变化。至少保留或填写 `name`、`category`、`default_unit`、`default_storage`、`default_expiry_mode`，否则先读取详情或追问。
- 例如“把番茄默认保质期改成 7 天”应先读取番茄详情，再生成 `action=update`，payload 中保留番茄现有名称、分类、主单位、保存位置和过期模式，并只把 `default_expiry_days` 改为 7。
- 已有库存时不得把主单位改成其他单位；该限制由后端再次校验。
- 更新食材时，除非用户明确要求更换或移除图片，否则不要改动图片相关字段；如果无法确认原媒体 ID，不要用空 `media_ids` 表达“保留原图”，应说明需要用户在审批表单里核对。
- 更新食材时没有明确换图、删图或替换媒体的要求，就不要传 `media_ids`；空数组可能被解释为清空媒体。
- 媒体、单位换算、保质期和库存提醒都必须使用真实字段，不编造业务 ID。
- 用户明确要求把上次确认的单位换算保存为副单位时，必须先读取真实食材详情，再调用 `ingredient_profile.create_draft` 生成单独更新审批；不要假装已保存。
- 从库存单位换算或菜谱缺食材流程转入本 Skill 时，优先复用前一轮已经确认的食材名称、单位、换算比例和候选信息；缺少关键字段时再追问，不要重新猜测。
- 用户确认前不得写入正式 `Ingredient`。
