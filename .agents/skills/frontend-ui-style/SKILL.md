# Culina 前端 UI 风格 Skill

> 本 Skill 是 Culina 产品内页的设计与实现规范，不是通用漂亮模板，也不是营销落地页 Skill。
> 目标是在不改变 Culina 当前主风格的前提下，让界面更清晰、更耐用、更统一、更像真实家庭厨房产品。

---

## 0. 执行原则

生成或修改 UI 前，必须先读取相关现有实现，再做设计判断。不要脱离项目现状重画一套风格。

优先参考：

* `frontend/src/styles/00-foundation.css`
* `frontend/src/styles/07-mobile.css`
* 对应业务域样式：

  * `01-home-dashboard.css`
  * `02-family-settings.css`
  * `03-recipe-workspace.css`
  * `04-ingredients-workspace.css`
  * `05-workspace-overlays.css`
  * `06-food-workspace.css`
  * `08-meal-log.css`
* 通用组件：`frontend/src/components/ui-kit.tsx`
* 应用壳：`frontend/src/app/AppShell.tsx`
* AI 页面：`frontend/src/components/ai/*`

如果通用 Skill、开源示例、模型默认审美与本 Skill 冲突，以本 Skill 为准。

## 1. Culina UI Read

动手前先在心里完成一次 UI 判断，必要时在回复里用一句话说明：

```txt
Reading this as: Culina 的家庭厨房产品内页，目标是提高任务清晰度和日常可用性，保留温暖家庭化主风格，不做营销页化重设计。
```

判断重点：

* 这是产品内页，不是 Landing Page。
* 用户是在完成真实任务：记录、补充、处理、安排、确认、复核。
* UI 应该先清楚、稳定、好维护，再追求“惊艳”。
* 不要为了好看牺牲信息层级、状态表达和触控可用性。
* 不要把页面改成通用 SaaS 后台、企业管理台、科技大屏或营销官网。

## 2. 产品气质

Culina 的视觉关键词：

* 温暖
* 家庭化
* 照片驱动
* 低维护
* 清晰任务入口
* 有轻微治愈感，但不是儿童化
* 有工具效率，但不是企业后台

文案默认简体中文，语气具体、克制、像家庭厨房助手。

推荐文案风格：

```txt
今天还没吃早餐
优先处理这些临期食材
确认后再写入计划
建议尽快补齐
库存状态平稳
```

避免文案风格：

```txt
开启你的智能饮食新体验
重塑家庭厨房工作流
AI 赋能下一代膳食管理
一键革新你的生活方式
```

AI 是实用助手，不是开放式泛聊天。AI 页面要体现：

* 家庭上下文
* 工具调用进度
* 草稿生成
* 用户审批确认
* 重要信息核对提醒
* 不直接承诺写入正式数据

## 3. 布局方式

### 3.1 桌面端

桌面端使用：

```txt
app-shell + app-frame + app-content
```

大屏结构是：

```txt
左侧 sidebar + 右侧工作区
```

内容最大宽度来自：

```css
--content-width
```

业务工作区优先使用 CSS Grid：

```css
display: grid;
gap: 14px / 16px / 18px / 22px / 24px;
```

列表优先使用：

```css
repeat(auto-fit, minmax(...))
```

详情、编辑、管理类页面优先使用：

```txt
主栏 + 侧栏
```

页面头优先复用：

* `PageHeader`
* `SectionHeading`
* `WorkspaceSubpageHeader`

不要重新发明页面头部结构。

复杂业务域使用局部 shell，例如：

* `recipe-workspace`
* `ingredients-workspace`
* `ai-workspace-shell`
* `food-workspace`
* `meal-log-workspace`

所有业务样式都应有明确业务前缀，避免污染全局。

### 3.2 移动端

手机端不是桌面压缩版。

已有移动视图应继续沿用：

* `HomeMobileDashboard`
* `FoodMobileView`
* `RecipeMobileLibraryView`
* `IngredientMobileView`
* `MealLogMobileView`
* `FamilyMobileView`
* `AiMobilePage`

移动端页面宽度通常限制为：

```css
width: min(100%, 480px);
margin: 0 auto;
```

底部要处理：

```css
env(safe-area-inset-bottom)
```

移动端关键原则：

* 独立信息架构
* 触控目标尽量不低于 44px
* 不横向溢出
* 不强行塞桌面两栏
* 底部导航、composer、烹饪底栏都要留安全区

## 4. 颜色体系

基础色来自 `:root` 变量。

当前主风格必须保留：暖白、米色、橙色、柔和绿色、奶油黄、浅棕边框。

### 4.1 基础色

```css
--bg: #FAF8F5;
--text: #2f251e;
--text-soft: #6A5B51;
--text-faint: #928378;

--accent: #D26B33;
--accent-strong: #B35122;
--accent-soft: #FCE1D4;

--sage-soft: #E2ECD8;
--butter-soft: #FDF3DB;

--line-soft: rgba(147, 96, 63, 0.12);
--line-strong: rgba(147, 96, 63, 0.22);
```

允许在不破坏主风格的前提下微调或新增语义变量，但不要引入另一套颜色系统。

推荐补充语义色：

```css
--surface: #FFFFFF;
--surface-warm: #FCFAF7;
--surface-muted: #F6F0EA;

--success: #4F9A58;
--success-soft: #EAF6EA;

--warning: #E69A2E;
--warning-soft: #FFF3D7;

--danger: #D94B3D;
--danger-soft: #FDEBE8;

--plan: #7B6FD6;
--plan-soft: #F0EEFF;
```

### 4.2 颜色使用规则

橙色是品牌主色，但不能到处使用。

橙色优先用于：

* 页面主 CTA
* 当前导航项
* 当前 tab
* 关键确认按钮
* 当前选中的日期 / 筛选项
* 需要用户立即处理的主操作

橙色不应滥用于：

* 所有小按钮
* 所有状态徽标
* 所有图标背景
* 所有边框
* 所有 hover
* 所有辅助链接

次级按钮建议使用：

```txt
白底 / 暖白底 + 浅棕边框 + 深色文字
```

而不是继续使用实心橙。

状态色语义固定：

```txt
成功 / 已完成 / 库存正常：柔和绿色
提醒 / 临期 / 待处理：黄橙
危险 / 已过期 / 失败：克制红
计划 / 推荐 / AI 辅助：柔和紫或浅橙紫
主操作：品牌橙
```

危险信息必须比普通提醒更明确。例如“已过期82天”应使用浅红底 + 红色文字，不要只用浅橙或浅粉导致警示感不足。

### 4.3 背景和表面

页面背景保持暖白，但需要和卡片拉开层级：

```txt
页面背景：浅米白 / 暖白
主要卡片：白色或近白
次级区域：浅暖白
边框：浅棕灰
阴影：暖棕低透明
```

不要让背景、卡片、边框、阴影全部都偏同一种奶油色，否则页面会“糊”。

### 4.4 禁止项

不要引入：

* 冷灰后台风
* 科技蓝主按钮
* 赛博紫大面积渐变
* 霓虹发光
* 纯黑暗色主题
* 高饱和撞色
* 和饮食家庭语境无关的科技感配色

## 5. 首页颜色和层级专项规则

首页是 Culina 的工作台，不是营销首屏。

首页优化方向：

* 保留温暖家庭感
* 减少橙色泛滥
* 增强卡片与页面背景区分
* 强化任务优先级
* 让主 CTA 更突出，次要按钮更安静

### 5.1 顶部统计卡

四个统计卡应保持柔和语义色：

```txt
在库食材：浅绿
临期提醒：浅桃 / 浅橙红
待采购：浅黄
本周做菜：浅紫
```

图标背景可以有淡色，但卡片本身尽量保持白底或近白底，避免一整块区域太花。

### 5.2 今天吃什么

“开始做”是当前推荐卡的主操作，可以使用实心橙。

列表 / 日历小按钮属于辅助操作，应使用：

```txt
白底 + 浅边框 + 深色图标
```

不要和“开始做”抢主次关系。

“换一批”是次级操作，优先使用：

```txt
白底 + 浅棕边框 + 深色文字
```

而不是实心橙。

### 5.3 临期优先处理

临期列表要表达风险等级：

* 已过期：浅红底 + 红字
* 7 天内到期：浅黄橙底 + 棕橙字
* 正常提醒：浅绿或浅米底

“去处理”可以保持橙色，但比主 CTA 稍弱，不要比页面顶部“新增食材”更抢眼。

### 5.4 本周菜单

当前日期卡：

```txt
浅暖底 + 橙色描边 + 橙色关键数字
```

不要使用过重填充。

未选中日期卡：

```txt
白底 + 浅边框 + 深色数字 + 弱化日期
```

状态汇总中：

* 已安排：浅橙
* 待补充：浅黄
* 已完成：浅绿

不要让所有状态都变成橙色。

## 6. 尺寸、排版与视觉细节规范（字体、间距、圆角、阴影、边框）

本章用于统一 Culina 前端在桌面端、平板端、弹窗、抽屉、表单、卡片和操作控件中的字体大小、元素尺寸、间距、圆角、阴影和边框风格。

整体原则：

* 保持 Culina 当前温暖、轻量、圆润、克制的主风格。
* 所有尺寸必须有体系，避免同一页面出现随机字体、随机圆角、随机间距。
* 桌面端强调信息密度与清晰层级；iPad / 平板端强调触控舒适度；移动端强调纵向流畅阅读。
* 弹窗和抽屉是重点统一对象，禁止同一业务域内出现大小、字体、按钮高度明显不一致的弹窗。

---

### 6.1 字体与排版层级规范（Typography Scale）

字体大小应基于 `1rem = 16px` 的系统基准。组件样式中禁止随意使用未归档的奇怪字号，例如 `11px`、`13px`、`17px`、`22px`、`32px` 等。

允许使用以下字号阶梯：

```css
--text-xs: 0.75rem;      /* 12px */
--text-sm: 0.8125rem;    /* 13px */
--text-meta: 0.875rem;   /* 14px */
--text-body: 0.9375rem;  /* 15px */
--text-base: 1rem;       /* 16px */
--text-md: 1.125rem;     /* 18px */
--text-lg: 1.25rem;      /* 20px */
--text-xl: 1.5rem;       /* 24px */
--text-2xl: 1.75rem;     /* 28px */
```

#### 页面级标题

* **页面主标题 / H1**

  * 字号：`1.5rem` ~ `1.75rem`
  * 字重：`700`
  * 行高：`1.25` ~ `1.3`
  * 使用场景：Dashboard 标题、一级页面标题、工作台主标题。

* **页面说明文字 / Page Description**

  * 字号：`0.9375rem` ~ `1rem`
  * 字重：`400` / `500`
  * 行高：`1.5`
  * 颜色应弱于标题，不要和主标题抢层级。

#### 卡片与区块标题

* **区块 / 卡片大标题 / H2**

  * 字号：`1.125rem` ~ `1.25rem`
  * 字重：`600`
  * 行高：`1.35` ~ `1.4`

* **次级卡片 / 列表分组标题 / H3**

  * 字号：`1rem`
  * 字重：`600`
  * 行高：`1.4`

* **小型模块标题 / Compact Title**

  * 字号：`0.9375rem`
  * 字重：`600`
  * 行高：`1.4`

#### 正文、标签与辅助文字

* **正文 / 列表项内容 / 输入框文字**

  * 字号：`0.9375rem`，必要时可用 `0.9rem`
  * 字重：`400` / `500`
  * 行高：`1.5`

* **表单 Label**

  * 字号：`0.875rem` ~ `0.9rem`
  * 字重：`600`
  * 行高：`1.4`

* **辅助说明 / 元信息 / 卡片副文案 / Badge**

  * 字号：`0.8125rem` ~ `0.875rem`
  * 字重：`400` / `500`
  * 行高：`1.4`

* **超小标签 / Eyebrow / 装饰性说明**

  * 字号：`0.75rem`
  * 字重：`500` / `600`
  * 行高：`1.3`
  * 可以使用轻微字间距，例如 `letter-spacing: 0.02em`。

### 6.2 弹窗与抽屉尺寸规范（Modal & Drawer）

所有 `WorkspaceModal`、`WorkspaceDrawer`、确认弹窗、快速操作弹窗必须遵循统一尺寸体系。禁止针对每个弹窗随意写一套宽度、字号、按钮高度。

#### 弹窗宽度

桌面端推荐宽度：

```css
--modal-width-sm: 440px;
--modal-width-md: 580px;
--modal-width-lg: 820px;
--modal-width-xl: 960px;
```

使用规则：

* **轻量确认 / 删除 / 警告弹窗**

  * `max-width: 440px`
  * 内容必须短，按钮不超过 2 个。

* **快速操作弹窗**

  * `max-width: 480px` ~ `540px`
  * 适用于“开始做这道菜”“快速记录”“确认日期和餐次”等轻流程操作。
  * 在 iPad 横屏中不得小于 `480px`，避免呈现为手机弹窗。

* **普通表单 / 中等任务弹窗**

  * `max-width: 560px` ~ `620px`
  * 适用于新增食材、新建分类、简单菜谱信息编辑。

* **复杂表单 / 详情编辑弹窗**

  * `max-width: 760px` ~ `860px`
  * 适用于菜谱编辑、餐食记录、含图片上传和多区块内容的弹窗。

* **大信息量工作流弹窗**

  * `max-width: 920px` ~ `960px`
  * 仅用于复杂 AI 结果确认、多列表并排、复杂数据预览。
  * 不得滥用大弹窗。

#### 弹窗响应式规则

```css
.workspace-modal {
  width: min(var(--modal-width-md), calc(100vw - 32px));
}

@media (min-width: 768px) {
  .workspace-modal {
    width: min(var(--modal-width-md), calc(100vw - 64px));
  }
}

@media (min-width: 1024px) {
  .workspace-modal {
    width: min(var(--modal-width-md), calc(100vw - 96px));
  }
}
```

iPad / 平板端特别规则：

* 快速操作弹窗建议宽度：`500px` ~ `560px`。
* 弹窗内按钮、日期选择、餐次选择必须优先保证触控舒适。
* 不要把手机端 `360px` 左右的弹窗直接放到 iPad 横屏中使用。
* iPad 横屏下弹窗视觉上应像“平板弹窗”，而不是“居中的手机卡片”。

#### 弹窗内边距

```css
--modal-padding-sm: 20px;
--modal-padding-md: 24px;
--modal-padding-lg: 28px;
```

使用规则：

* 轻量确认弹窗：`20px` ~ `24px`
* 普通弹窗：`24px`
* 复杂弹窗：`24px` ~ `28px`
* 移动端弹窗可降至 `18px` ~ `20px`
* 内容密集时优先增加分组间距，不要压缩文字行高。

#### 弹窗字体

* **弹窗标题**

  * 字号：`1.125rem` ~ `1.25rem`
  * 字重：`600` / `700`
  * 行高：`1.35`
  * 禁止使用 `1.5rem` 以上的大标题，除非是欢迎页级别弹窗。

* **弹窗副标题 / Eyebrow**

  * 字号：`0.75rem` ~ `0.8125rem`
  * 字重：`500` / `600`
  * 颜色弱于标题。

* **弹窗正文**

  * 字号：`0.9375rem`
  * 行高：`1.5`

* **弹窗内区块标题**

  * 字号：`1rem`
  * 字重：`600`
  * 行高：`1.4`

* **弹窗表单 Label**

  * 字号：`0.875rem`
  * 字重：`600`

* **弹窗辅助说明**

  * 字号：`0.8125rem`
  * 行高：`1.4`

* **弹窗底部按钮文字**

  * 字号：`0.9375rem`
  * 字重：`600`

### 6.3 触控热区与操作控件尺寸规范

Culina 有家庭厨房使用场景，用户可能在 iPad、手机或触控屏上操作。因此所有高频操作控件必须保证点击舒适度。

#### 基础触控热区

* 所有可点击元素的最小点击热区不得小于 `40px × 40px`。
* iPad / 移动端高频按钮建议不小于 `44px × 44px`。
* 关闭按钮、返回按钮、图标按钮即使视觉图标较小，也必须保留足够热区。

```css
--tap-min: 40px;
--tap-comfortable: 44px;
--tap-large: 48px;
```

#### 按钮高度

* **桌面普通按钮**

  * 高度：`40px` ~ `44px`

* **iPad / 平板按钮**

  * 高度：`44px` ~ `48px`

* **移动端主按钮**

  * 高度：`48px` ~ `52px`

* **轻量工具按钮**

  * 高度：`36px` ~ `40px`
  * 仅限工具栏、筛选栏、卡片内次级操作。

* **弹窗底部主按钮**

  * 高度：`46px` ~ `52px`
  * 主按钮必须比普通文字链接更明显。

#### 图标按钮

```css
.icon-button {
  width: 40px;
  height: 40px;
}

@media (min-width: 768px) {
  .icon-button {
    width: 44px;
    height: 44px;
  }
}
```

规则：

* 图标本身可以是 `18px` ~ `22px`。
* 不能只给图标本身设置点击事件，必须让外层按钮承载点击热区。
* 弹窗右上角关闭按钮建议使用 `40px` ~ `44px` 热区。

---

### 6.4 表单元素与选择控件尺寸规范

#### 输入框 / 选择框

* 桌面端高度：`42px` ~ `44px`
* iPad / 平板端高度：`44px` ~ `48px`
* 移动端高度：`46px` ~ `50px`

```css
--input-height-sm: 40px;
--input-height-md: 44px;
--input-height-lg: 48px;
```

规则：

* 同一弹窗内输入框高度必须一致。
* 输入框圆角建议 `12px` ~ `14px`。
* 输入框内部左右 padding 建议 `12px` ~ `14px`。
* Textarea 不固定高度，但最小高度建议 `96px`。

#### 日期 / 餐次 / Chip 类选择器

日期选择、餐次选择、分类 Chip 是 Culina 的高频交互控件，不能做得过小。

* 日期卡片：

  * 桌面端最小：`52px × 52px`
  * iPad / 平板端建议：`56px × 56px`
  * 移动端可根据屏宽自适应，但点击热区不得小于 `44px`

* 餐次按钮：

  * 高度：`42px` ~ `48px`
  * 圆角：`999px`
  * 多个按钮并排时应等宽或接近等宽。

* 普通 Chip：

  * 高度：`32px` ~ `36px`
  * 高频选择 Chip 可提高到 `38px` ~ `42px`

#### 快速操作弹窗示例规则

类似“开始做这道菜”这种弹窗：

```css
.quick-action-modal {
  width: min(520px, calc(100vw - 64px));
  padding: 24px;
  border-radius: 24px;
}

.quick-action-date {
  min-width: 56px;
  height: 56px;
  border-radius: 14px;
}

.quick-action-meal {
  height: 46px;
  border-radius: 999px;
}

.quick-action-footer button {
  height: 50px;
}
```

### 6.5 间距规范（Spacing）

Culina 的空间感应保持“松弛但不空”的状态。页面不要过度拥挤，也不要为了高级感而留白过大。

推荐间距 Token：

```css
--space-1: 4px;
--space-2: 6px;
--space-3: 8px;
--space-4: 12px;
--space-5: 16px;
--space-6: 20px;
--space-7: 24px;
--space-8: 28px;
--space-9: 32px;
```

使用规则：

* **文字与关联说明之间**

  * `4px` ~ `6px`

* **Label 与输入框之间**

  * `6px` ~ `8px`

* **表单项之间**

  * `14px` ~ `18px`

* **卡片内部元素之间**

  * `12px` ~ `16px`

* **卡片内边距**

  * 紧凑卡片：`16px` ~ `18px`
  * 普通卡片：`20px` ~ `24px`
  * 重点卡片：`24px` ~ `28px`

* **页面主区域 Grid Gap**

  * 桌面端：`22px` ~ `24px`
  * 平板端：`18px` ~ `22px`
  * 移动端：`14px` ~ `18px`

* **弹窗内容分组之间**

  * `18px` ~ `24px`

* **弹窗 Header 与 Body 之间**

  * `16px` ~ `20px`

* **弹窗 Body 与 Footer 之间**

  * `20px` ~ `24px`

---

### 6.6 圆角规范（Radius）

Culina 的圆角风格应保持温和、圆润，但不能所有元素都无限圆角。不同组件应有明确层级。

推荐圆角 Token：

```css
--radius-xs: 10px;
--radius-sm: 14px;
--radius-md: 18px;
--radius-lg: 24px;
--radius-xl: 28px;
--radius-pill: 999px;
```

使用规则：

* **输入框 / Select / 小按钮**

  * `10px` ~ `14px`

* **普通按钮**

  * `12px` ~ `14px`

* **日期卡片 / 小选择卡**

  * `12px` ~ `16px`

* **基础卡片**

  * `18px` ~ `22px`

* **业务重点卡片**

  * `20px` ~ `24px`

* **大容器 / 工作台面板**

  * `24px` ~ `28px`

* **弹窗**

  * `22px` ~ `28px`

* **胶囊按钮 / Chip / Tab**

  * `999px`

禁止项：

* 禁止同一页面内出现 `8px`、`13px`、`17px`、`21px`、`31px` 等无体系圆角。
* 禁止所有元素统一使用 `999px`，只有胶囊类组件可以使用。
* 禁止按钮圆角大于外层卡片圆角，除非按钮是胶囊样式。

---

### 6.7 阴影规范（Shadow）

Culina 的阴影必须轻、暖、柔和，禁止使用纯黑高透明重阴影。

推荐阴影：

```css
--shadow-xs: 0 4px 12px rgba(74, 54, 40, 0.06);
--shadow-sm: 0 8px 24px rgba(74, 54, 40, 0.08);
--shadow-md: 0 14px 36px rgba(74, 54, 40, 0.10);
--shadow-lg: 0 22px 56px rgba(74, 54, 40, 0.14);
```

使用规则：

* 普通卡片：`--shadow-xs` 或不用阴影，仅使用浅边框。
* 悬浮卡片 / Dropdown：`--shadow-sm`
* 弹窗：`--shadow-md` ~ `--shadow-lg`
* 重点悬浮层：允许使用更大 blur，但透明度不得过重。

禁止项：

* 禁止使用 `rgba(0, 0, 0, 0.25)` 这类黑色重阴影。
* 禁止多个重阴影叠加造成脏灰感。
* 禁止 hover 时阴影突然大幅增强。

---

### 6.8 边框规范（Border）

Culina 的边框应轻、暖、低对比，用于建立结构，不应形成冰冷的灰色线框感。

推荐边框颜色：

```css
--line-soft: rgba(92, 67, 48, 0.08);
--line-normal: rgba(92, 67, 48, 0.12);
--line-strong: rgba(92, 67, 48, 0.16);
```

使用规则：

* 普通卡片边框：`1px solid var(--line-soft)`
* 输入框默认边框：`1px solid var(--line-normal)`
* 输入框 focus：边框变暖，并增加轻微外发光。
* 选中态卡片 / Chip：使用品牌橙或暖橙浅色边框。
* 分割线透明度要低，不能像后台管理系统一样硬。

---

### 6.9 Hover、Active 与动效规范

Culina 的动效应轻微、自然、稳定，不能炫技。

#### 卡片 Hover

允许：

```css
.card:hover {
  transform: translateY(-1px);
  box-shadow: var(--shadow-sm);
  border-color: rgba(213, 112, 55, 0.22);
}
```

规则：

* 卡片最多上浮 `1px` ~ `2px`
* 阴影只能轻微增强
* 边框可略微变暖
* 不要出现明显缩放、旋转、闪烁、弹跳。

#### 按钮 Active

允许：

```css
.button:active {
  transform: translateY(1px);
}
```

规则：

* 点击反馈应短促、轻微。
* 主按钮 hover 可以略微加深背景色。
* 次级按钮 hover 可以增强边框或背景，不要抢主按钮层级。

#### 动效时间

```css
--motion-fast: 120ms;
--motion-normal: 180ms;
--motion-slow: 240ms;
```

使用规则：

* Hover：`120ms` ~ `180ms`
* 弹窗进入：`180ms` ~ `240ms`
* 抽屉进入：`220ms` ~ `280ms`
* 禁止超过 `400ms` 的拖沓动效。

---

### 6.10 禁止项与检查清单

#### 禁止项

* 禁止同一页面出现多个无规律字号。
* 禁止弹窗标题过大，尤其禁止普通弹窗使用页面级 H1。
* 禁止 iPad 横屏下继续使用过窄的手机弹窗。
* 禁止日期、餐次、关闭按钮等高频点击元素小于可舒适点击尺寸。
* 禁止使用纯黑重阴影。
* 禁止边框过灰、过硬、过后台系统感。
* 禁止同一页面随机出现大量不同圆角。
* 禁止为了“精致”把按钮、输入框、日期选择做得过小。
* 禁止 hover 出现夸张位移、缩放、旋转或复杂动画。

#### 开发前检查

在新增或修改 UI 时，必须检查：

* 字号是否来自本章 Typography Scale。
* 弹窗宽度是否匹配任务复杂度。
* iPad / 平板端是否有足够触控热区。
* 按钮、输入框、选择器高度是否统一。
* 卡片、弹窗、按钮圆角是否符合层级。
* 阴影是否为暖棕低透明，而不是纯黑重阴影。
* 边框是否轻、暖、低对比。
* 同一业务域内弹窗是否保持一致。

#### 推荐结论

当不确定尺寸时，优先选择以下默认值：

```css
/* 字体 */
--default-body-size: 0.9375rem;
--default-title-size: 1.125rem;

/* 卡片 */
--default-card-padding: 20px;
--default-card-radius: 20px;

/* 弹窗 */
--default-modal-width: 580px;
--default-modal-padding: 24px;
--default-modal-radius: 24px;

/* 操作控件 */
--default-button-height: 44px;
--default-input-height: 44px;
--default-touch-size: 44px;

/* 平板快速弹窗 */
--tablet-quick-modal-width: 520px;
--tablet-date-option-size: 56px;
--tablet-action-button-height: 50px;
```

新增视觉元素前先判断：

```txt
是否能用已有组件组合？
是否只是已有组件的业务变体？
是否真的需要新增组件？
是否会被其他页面复用？
```

## 8. 卡片

卡片通常由以下内容组成：

```txt
暖白表面
浅棕边框
轻阴影
照片 / 图标
标题
元信息 chip
状态
行动按钮
```

食物、菜谱、食材卡片优先展示真实图片或已有占位图。

图片规则：

```css
aspect-ratio: ...;
object-fit: cover;
```

占位图应使用暖色、绿色、浅米色柔和渐变，不要使用抽象 SVG 营销插画。

长标题必须处理：

```css
min-width: 0;
overflow: hidden;
text-overflow: ellipsis;
overflow-wrap: anywhere;
```

卡片内按钮主次要明确：

* 主操作：实心橙
* 次操作：白底描边
* 轻操作：纯文本或图标按钮

不要一个卡片里出现多个同等强度的橙色按钮。

## 9. 表单

基础输入使用 `.text-input`。

表单结构：

```txt
label
input / control
helper text
error text
```

不要用 placeholder 代替 label。

两列表单用 `.form-grid`，移动端改为单列。复杂表单拆分为：

```txt
.form-panel-section
```

数量、库存、评分、份量等触控输入优先使用：

* `TouchStepperField`
* `TouchRangeField`
* chip 快捷值

触控控件尽量接近或不低于 44px。

按钮规则：

* 提交按钮：`solid-button`
* 次要操作：`ghost-button`
* 文本轻操作：`tertiary-button`

提交中、禁用、错误文案必须明确。
不要只靠颜色表达状态。

---

## 10. 弹窗和抽屉

优先使用：

* `WorkspaceModal`
* `WorkspaceDrawer`

保持结构：

```txt
workspace-overlay-panel
workspace-overlay-head
workspace-overlay-body
workspace-overlay-actions
```

弹窗标题区使用：

```txt
eyebrow + h3 + subtle 描述
```

关闭按钮必须保留：

```txt
aria-label
```

或屏幕阅读文本。

### 10.1 弹窗尺寸与排版规范

* **尺寸与排版一致性**: 弹窗内所有字体、间距、高度及最大宽度必须严格遵循 [6.2 弹窗（Modal）与抽屉（Drawer）尺寸与字体一致性规范](#62-弹窗modal与抽屉drawer尺寸与字体一致性规范)。
* **避免硬编码**: 禁止在各个弹窗样式文件中直接使用硬编码像素值，所有新写的或修改的弹窗样式都必须在统一的排版阶梯指导下声明。
* **同一页面一致性**: 同一个业务模块（例如食材库、菜谱库、餐食记录等）下，所有弹窗与抽屉的字体大小、行高、输入框高度和动作按钮必须完全一致，严禁同一个页面内的不同弹窗出现视觉规格断层。

### 10.2 业务表达与移动端适配

重要确认类弹窗必须清楚展示：

* 操作对象
* 影响范围
* 是否会写入正式数据
* 主按钮
* 次按钮

不要只弹一句泛提示。

移动端弹窗必须检查：

* 宽度
* 底部安全区
* 滚动区
* 按钮是否可触达
* 键盘弹出后的可用性

## 11. 列表、状态栏和导航

标签、状态、筛选项常用胶囊 chip：

```css
border-radius: 999px;
min-height: 30px-44px;
```

当前状态使用橙色或业务状态色。
非当前项使用暖白 / 浅米色。

桌面导航：

```txt
sidebar + 顶部 tabbar / subnav
```

手机导航：

```txt
底部导航
```

AI tab 可以保持头像式突出入口，但不要让它破坏整体导航节奏。

工作区子导航优先使用：

* `WorkspaceSubnav`
* 同等结构

不要新增横向密集小字导航。

状态栏要表达：

```txt
正在做什么
已经完成什么
下一步是什么
是否需要用户确认
```

例如 AI 运行进度、库存状态、餐食记录状态。

状态栏默认应是摘要态，必要时可展开查看详细进度。
不要默认展开一大串过程，占据主要阅读空间。

## 12. AI 对话页

AI 桌面页使用：

```txt
ai-workspace-shell
```

结构：

```txt
左侧历史 / 上下文
右侧主对话面板
```

主面板：

```txt
暖白渐变
圆角 26px
固定高度内滚动
```

AI 移动页使用：

* `AiMobilePage`
* `AiMobileChrome`

不要直接复用桌面两栏。

### 12.1 AI 配色

AI 页面要比其他页面更克制，因为阅读量大。

建议：

* 聊天背景使用更干净的暖白
* 渐变透明度降低
* 不要大面积橙色发光
* 用户气泡可以是浅橙暖渐变
* 助手消息使用白色卡片
* 工具调用 chip 使用中性暖灰或浅橙描边
* 完成状态用绿色
* 错误状态用红色
* 暂停 / 等待确认用黄橙

不要把整个 AI 页面做成满屏暖光或营销 hero。

### 12.2 AI 消息

消息气泡：

* 助手：左侧头像 + 白色卡片
* 用户：右侧橙色暖渐变气泡
* 审批表单：助手消息可更宽
* 重要草稿：使用结构化卡片，不只是一段长文本

必须保留：

* `ai-run-progress`
* tool chip
* 审批面板
* composer 暂停态
* 发送 / 中止状态
* 免责声明

欢迎态可以有 AI 厨师 / 机器人图片和快捷问题，但不要写成营销首屏。

### 12.3 AI 文案

AI 文案避免：

* 医疗诊断
* 营养治疗承诺
* 绝对化建议
* 未确认就写入计划
* 夸张营销表达

推荐表达：

```txt
我会先生成草稿，确认后再写入正式计划。
请核对食材数量和日期。
这只是家庭饮食安排建议，不替代专业医疗或营养建议。
```

## 13. 动效和交互

Culina 的动效应该轻、短、实用。

允许：

* hover 轻微上浮
* 按钮 active 轻微下压
* 弹窗淡入
* 抽屉滑入
* 列表项轻微过渡
* AI 状态从“执行中”到“完成”的状态切换

避免：

* 大幅缩放
* 炫技滚动动画
* 复杂视差
* 无限循环装饰动画
* 鼠标磁吸
* 自定义光标
* 大面积玻璃拟态

动效必须有目的：

```txt
反馈用户操作
表达状态变化
引导注意力
让层级变化更自然
```

如果说不清动效在表达什么，就不要加。

必须尊重：

```css
prefers-reduced-motion
```

高频输入、滚动、鼠标位置不要用 React state 每帧更新。
能用 CSS transition 就不要引入复杂动画库。

---

## 14. 可访问性和可用性

按钮文字必须能看清。

禁止：

```txt
白底白字
浅橙底白字但对比不足
透明按钮没有边框
图片背景上的 ghost button 没有遮罩
```

按钮文字桌面端不能换行。
如果按钮文案太长：

* 缩短文案
* 增加按钮宽度
* 改成说明文字 + 短按钮

表单必须有 label。
错误信息必须有文字。
状态不能只靠颜色表达。

移动端触控目标尽量不低于 44px。

图片必须有稳定尺寸，避免 CLS。
头像、菜品图、食材图必须有占位策略。

---

## 15. 文案规则

文案要短、具体、直接。

推荐：

```txt
去处理
开始做
补食材
确认草稿
查看记录
换一批
今天还没吃早餐
3 条提醒待处理
```

避免：

```txt
立即开启
探索更多
智能生成美好生活
赋能厨房管理
全新体验
```

长文案要拆成结构化信息，不要堆一大段。

AI 回复尤其要避免：

* 空泛
* 过度拟人
* 强营销
* 自称能力过大
* 没有确认就承诺执行

---

## 16. 移动端适配

主要移动层：

```css
@media (max-width: 767px)
```

移动端必须检查：

* 375px
* 390px
* 430px
* iOS 底部安全区
* 键盘弹出
* 底部导航遮挡
* 长标题换行
* chip 换行
* 横向滚动
* 弹窗按钮是否露出

移动端标题可更紧凑，但不要用 viewport 宽度缩放字体。

移动端卡片、表单、chip 需要允许换行或单列，避免横向溢出。

关键路径改动后要用 smoke 或浏览器截图检查。

## 17. 代码和样式约束

不要新增全局裸标签样式，例如：

```css
button {}
input {}
.card {}
```

除非在 foundation 层明确需要。

业务样式必须有业务前缀，例如：

```css
.home-dashboard-*
.recipe-workspace-*
.ingredients-workspace-*
.food-workspace-*
.ai-*
.meal-log-*
```

不要绕开 `styles.css` 聚合入口。

不要在 TSX 里堆大量 inline style 表达业务 UI。

不要随意新增第三方 UI 库。
如果必须引入依赖，先检查 `package.json`，确认项目是否已有同类依赖。

一个业务区域不要混用多套图标风格。
图标尺寸、线宽、颜色要统一。

## 18. 常见禁止项

禁止：

* 通用 SaaS 后台管理系统风
* 冷灰表格
* 密集管理台
* 硬黑边框
* 蓝色主按钮
* 营销落地页 hero 替代真实应用第一屏
* 大面积 AI 紫渐变
* 霓虹发光
* 强玻璃拟态
* 厚重阴影
* 超大随机圆角
* 一次性颜色体系
* 抽象 SVG 营销插画
* 假截图式 div 装饰
* 把移动端当桌面 JSX 简单隐藏
* 文本、按钮、图片、状态徽标互相重叠
* 长文本无换行、省略或滚动策略
* 所有按钮都用橙色
* 所有状态都用橙色
* 所有卡片都靠阴影区分
* 为了“高级感”牺牲可读性

## 19. 页面修改前检查

修改 UI 前先检查：

```txt
1. 这个页面的主任务是什么？
2. 用户下一步最可能要点哪里？
3. 哪个按钮是主 CTA？
4. 哪些按钮应该降级为次级按钮？
5. 当前状态色是否符合语义？
6. 橙色是否用得过多？
7. 卡片和背景是否区分清楚？
8. 移动端是否需要独立结构？
9. 是否可以复用已有组件？
10. 是否会污染其他业务域样式？
```

## 20. 提交前 Pre-flight Check

每次完成 UI 修改前，必须自查：

### 视觉

* 是否保留 Culina 暖白、橙色、家庭厨房主风格？
* 是否减少了不必要的橙色？
* 主 CTA 是否明确？
* 次级按钮是否降级？
* 状态色是否语义一致？
* 卡片与页面背景是否有清晰层级？
* 圆角、边框、阴影是否统一？

### 可用性

* 按钮文字是否清楚？
* 按钮是否有足够对比度？
* 桌面端按钮文字是否单行？
* 长标题是否省略或换行？
* 图片是否有固定比例？
* 状态是否不只靠颜色表达？
* 错误、空状态、加载中是否有明确表达？

### 移动端

* 375px / 390px 是否无横向溢出？
* 底部安全区是否处理？
* 触控目标是否接近 44px？
* 弹窗、抽屉、composer 是否可触达？
* 底部导航是否遮挡内容？

### 代码

* 是否复用 ui-kit？
* 是否使用业务前缀？
* 是否避免大量 inline style？
* 是否没有新增冲突颜色体系？
* 是否没有绕开 `styles.css`？
* 是否没有引入不必要依赖？

## 21. 对不同任务的默认处理

### 新页面

先建立页面信息架构，再套用 Culina 视觉系统。
不要直接生成营销式大 hero。

### 改页面

优先保留现有结构，只优化：

* 层级
* 配色
* 间距
* 状态表达
* 响应式
* 可用性

除非用户明确要求重构，不要大改交互模型。

### 改组件

先检查是否影响多个页面。
通用组件改动要更谨慎，业务特有变化优先做局部变体。

### 改 AI 页面

优先保证：

* 对话可读性
* 工具进度清楚
* 审批确认明确
* 输入区状态明确
* 不被背景和渐变干扰

### 改首页

首页是任务工作台。
优先优化：

* 今日任务入口
* 临期处理
* 待采购
* 本周菜单
* 当前状态概览

不要把首页改成介绍产品价值的营销页。

## 22. 一句话总原则

Culina 的 UI 应该像一个温暖、清楚、可信赖的家庭厨房工作台：
主色克制，状态清楚，任务入口明确，移动端好用，AI 结果可确认。
