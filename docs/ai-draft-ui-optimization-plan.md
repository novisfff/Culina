# AI 审批 Draft UI 统一优化说明

本文用于把 AI 助手里的所有结构化 draft 审批界面，统一到 `ingredient_profile` 食材档案草稿目前采用的展示标准。

检查范围：

- 前端结构化编辑器：`frontend/src/components/ai/AiApprovalPanel.tsx`
- 通用审批字段：`frontend/src/components/ai/AiApprovalFields.tsx`
- 库存操作编辑器：`frontend/src/components/ai/AiInventoryOperationEditor.tsx`
- 复合操作预览：`frontend/src/components/ai/AiCompositeOperationPreview.tsx`
- 后端审批类型矩阵：`backend/app/services/ai_operations/approval_config.py`

## 统一标准

以食材档案 draft 为基准，所有 draft 都应满足以下标准。

1. 摘要优先：顶部先给用户一眼能判断的摘要，不要求先读完整表单。
2. 影响说明明确：说明确认后会写入什么、不会写入什么、是否会扣库存、是否会创建关联对象。
3. 分区编辑：复杂 draft 拆成“核心信息 / 影响与默认值 / 高级设置或补充信息”，避免一整块长表单。
4. 有限选项用下拉：枚举、状态、布尔值、类型、餐别、难度、库存状态等不允许随意输入；允许自定义时用可输入下拉。
5. 资源引用用选择器：食材、食物、计划、批次、成员、媒体等引用不能让用户手填 ID。
6. 数量和单位结构化：数量、单位、换算、份数要拆成稳定字段，必要时显示单位后缀。
7. 提交前校验：明显不合法的值在前端阻止确认，并给出具体错误。
8. 已处理状态压缩：approved / rejected / expired 不再展示完整禁用表单，改为紧凑摘要卡。
9. 移动端可用：触控目标接近 44px，菜单不使用原生系统浮层，长文案不横向溢出。

## 通用组件建议

为避免继续扩大 `AiApprovalPanel.tsx`，后续实现时建议拆出以下轻量组件：

- `AiDraftSummaryCard`：统一 pending 和 resolved 摘要卡。
- `AiDraftSection`：统一分区标题、说明和间距。
- `AiDraftImpactNote`：统一“确认后会/不会做什么”的说明块。
- `AiDraftResolvedSummary`：统一 approved / rejected / expired 压缩态。
- `AiDraftValidation` 或 model helper：把各 draft 的提交前校验从组件 JSX 中移出。
- `ApprovalTagInput`：替代顿号文本输入，用于口味标签、关键点、场景标签。
- `ApprovalUnitCombobox`：封装常用单位和自定义单位。

## 实施顺序

建议按风险和用户感知排序：

1. P0：`recipe`、`recipe_cook`、`meal_log`。这些 draft 写入影响最大，且当前和食材档案标准差距最明显。
2. P1：`meal_plan`、`shopping_list`、`food_profile`、`inventory_operation`。这些已有结构化编辑器，主要是摘要、分区、下拉和校验补齐。
3. P2：`composite_operation`。先做可读预览和风险表达，暂不急着实现子步骤可编辑。
4. 收尾：所有 draft 的 resolved 状态统一压缩摘要，减少禁用长表单。
