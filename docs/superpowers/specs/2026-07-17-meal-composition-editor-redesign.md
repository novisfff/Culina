# 餐食组合编辑器重设计规格

日期：2026-07-17

## 目标

重做“调整组合”弹窗，使用户只处理这顿饭包含哪些食物以及每个食物的备注。移除当前大表单卡片、菜品 ID 和份量编辑，改为图片驱动的紧凑清单，并使用现有食物库完成搜索添加。

本次保留现有保存、版本冲突、超时复核和再次确认语义，不修改餐食详情照片区、餐食评价或其他记录入口。

## 内容与操作

- 弹窗标题为“调整组合”，说明为“添加或移除这顿包含的食物”。
- 每个条目显示食物图片、名称、食物类型、备注输入和删除按钮。
- 食物名称只读；界面中不显示或编辑 `food_id`。
- 界面中不显示份量、份量输入、数字或加减控件。
- 已有条目的 `servings` 在保存 payload 中原样保留。
- 搜索新加入的食物固定使用 `servings: 1`，该默认值不在界面中展示。
- 删除至少保留一道食物；只剩一道时删除按钮禁用并提供可访问说明。
- 备注为空时使用“备注”或“添加备注”作为简短提示，保存时继续执行现有 trim。

## 搜索添加

- 弹窗正文底部常驻“搜索并添加食物”输入。
- 复用 `SearchableResourceSelect`、`SearchField`、媒体 URL 与系统图片占位能力。
- 搜索范围是 `MealLogWorkspace` 已加载的家庭食物列表，不新增请求。
- 按食物名称过滤，结果显示图片、名称和类型。
- 已经存在于当前餐食组合中的食物从结果中排除。
- 选择结果后立即追加本地条目，使用真实 `food_id`、食物名称和固定默认份量 1，并清空搜索词。
- 搜索不到时显示“没有找到相关食物”。本弹窗不创建新食物，也不提供自由文本或菜品 ID 回退。
- 图片无数据、加载中或加载失败时使用系统占位。

## 响应式视觉

### 桌面与 Pad

- 使用中型弹窗和紧凑清单，不使用每道菜一张大卡片。
- 清单列为“食物 / 备注 / 删除”。
- 食物区域显示约 42px 图片、名称和类型；名称过长时单行截断。
- 备注使用单行输入并获得剩余宽度。
- 删除使用安静的图标按钮，不使用文字“移除”。
- 搜索输入位于清单下方，结果以 popover 展示。

### 手机

- 使用现有移动端 bottom sheet 结构。
- 每个食物固定为单行布局：`36px 图片 / 名称与类型 / 备注 / 删除`。
- 单行目标高度约 52px，内容间距约 7px，长名称单行截断。
- 备注输入可见高度约 34px，并使用受控弹性宽度。
- 删除按钮保持 44px 实际命中区域；搜索入口可见高度约 38px。
- 在 320px 宽度下仍保持单行，不产生横向滚动。
- 底部取消和保存保持 44px 高，并处理安全区。

## 状态与保存

- 编辑器继续维护 base、draft、expected row version、冲突、提交中和错误状态。
- 搜索词和搜索开关只属于界面状态，不进入保存 payload。
- 提交中禁用搜索、备注、删除、关闭和重复提交。
- 保存失败后保留当前组合与备注。
- 409 响应继续执行现有三方合并，并要求用户显式再次保存。
- 超时后继续重新获取餐食；内容一致时视为成功，不一致时合并并要求确认。
- 冲突提示不再出现份量字段的用户文案；历史数据出现 servings 冲突时保留内部合并结果，但界面只提示“内容已变化，需要再次确认保存”。

## 实现边界

- `MealCompositionEditor.tsx`：重构可见结构、搜索状态、添加已有食物、删除和备注编辑。
- `MealLogWorkspace.tsx`：向编辑器传入当前家庭食物列表。
- `08-meal-log.css`：实现桌面、Pad 和手机布局，保持业务前缀并复用规范 token。
- `MealCompositionEditor.test.tsx`：覆盖搜索、添加、排除重复、不显示 ID/份量、备注、删除下限和现有保存恢复行为。
- `frontend/scripts/smoke.mjs`：覆盖手机 375×812、Pad 1024×768、桌面 1440×960 的弹窗结构、触控尺寸、搜索添加和横向溢出。

不修改后端 schema、API、数据库、餐食组成 payload 类型或媒体存储。

## 验收标准

- 页面中不存在“菜品 ID”“份量”及对应输入。
- 已有食物显示正确图片、名称、类型和备注。
- 搜索只返回尚未加入的现有食物，选择后立即追加。
- 保存时已有条目的 servings 保持原值，新条目 servings 固定为 1。
- 至少保留一道食物，禁用状态不只依赖颜色。
- 手机单个食物为一行，320px 起无横向溢出；桌面和 Pad 不恢复大卡片布局。
- 原有 409、超时复核、再次确认和错误保留测试继续通过。
- 定向 Vitest、`frontend:quality`、`frontend:build`、样式 token 检查和目标 smoke 均执行；既有无关失败单独报告。
