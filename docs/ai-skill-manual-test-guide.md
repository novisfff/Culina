# AI Skill 手动测试说明

更新时间：2026-06-16

## 1. 测试目标

本说明用于手动验证 AI Workspace 的 Skill 能力是否符合整改目标，重点测试：

- Skill 路由是否正确。
- Tool 是否只生成草稿，不直接写正式业务表。
- 审批面板是否显示正确动作和影响。
- 用户确认后正式业务数据是否正确变化。
- 用户拒绝、目标冲突、跨家庭或模糊目标是否安全处理。

## 2. 测试前准备

建议准备一个测试家庭，至少包含：

- 当前用户是家庭 Owner 或有 AI 使用权限的成员。
- 食材：番茄、鸡蛋、牛奶。
- 库存：番茄 3 个、鸡蛋 4 个。
- 食物：番茄炒蛋。
- 菜谱：番茄炒蛋，关联番茄和鸡蛋。
- 明天晚餐计划：一个可被修改的计划项。
- 购物清单：鸡蛋 2 盒，未完成。
- 一条最近餐食记录，包含可评分的食物项。

测试时打开 AI 工作台，优先使用自然语言输入，不要直接调用接口。

### 2.1 快速脚本执行

可以用 Playwright 脚本自动执行核心用例，脚本不会挂到构建流程，需要手动运行：

```bash
npm --prefix frontend run ai:skill-smoke -- --username=<测试账号> --password=<测试密码>
```

常用参数：

- `--url=http://127.0.0.1:5173`：使用已启动的前端；不传时脚本会自动启动 Vite dev server。
- `--backend=http://127.0.0.1:8010`：自动启动前端时使用的后端地址。
- `--decision=approve|reject|none`：默认 `approve`，会真实修改测试家庭数据；只想检查审批出现可用 `reject`。
- `--cases=core|destructive|all|a,b`：默认 `core`，删除类用例需显式运行 `destructive` 或 `all`。
- `--headed --slow-mo=250`：打开浏览器窗口并放慢操作，便于观察。

示例：

```bash
npm --prefix frontend run ai:skill-smoke -- --username=test-owner --password=test-password --decision=approve --headed
```

## 3. 通用验收规则

每个审批型任务都应满足：

1. AI 回复中出现待确认审批，而不是直接说已经写入。
2. 审批面板显示动作类型、目标对象和主要字段。
3. 点击拒绝后，正式业务页面不发生变化。
4. 点击确认后，正式业务页面发生预期变化。
5. 成功后出现操作结果卡。
6. 修改、删除、完成、恢复等目标绑定操作必须能看出目标是谁。

## 4. Skill 手动测试用例

### 4.1 ingredient_profile

输入：

```text
帮我新增一个食材：秋葵，默认单位按根，常温保存，不设置保质期。
```

预期：

- 路由到食材档案能力。
- 生成 `ingredient_profile` 审批。
- 审批面板显示“创建食材”。
- 确认后食材库出现“秋葵”。

继续输入：

```text
把秋葵的默认保存位置改成冷藏，备注写成焯水后更适合凉拌。
```

预期：

- AI 先定位真实“秋葵”食材。
- 生成更新审批，包含当前值和修改值。
- 确认后食材详情被更新。

### 4.2 inventory_analysis

输入：

```text
把今天买的鸡蛋 2 盒录入库存，放冷藏。
```

预期：

- AI 匹配当前家庭已有“鸡蛋”食材。
- 生成库存入库审批。
- 确认后库存增加新批次。

输入：

```text
今天用了 2 个番茄。
```

预期：

- 生成库存消耗审批。
- 确认后番茄库存剩余量减少。

输入：

```text
把已经过期的番茄销毁。
```

预期：

- AI 读取可销毁批次。
- 若目标不明确，要求用户确认批次。
- 确认后对应批次 disposed quantity 增加。

### 4.3 food_profile

输入：

```text
新增一个食物资料：盒装牛奶，类型是即食，适合早餐。
```

预期：

- 生成 `food_profile` 创建审批。
- 确认后食物库出现“盒装牛奶”。

输入：

```text
把盒装牛奶加入常用收藏。
```

预期：

- AI 定位真实食物。
- 生成收藏状态审批。
- 确认后食物收藏状态变为 true。

### 4.4 recipe_draft

输入：

```text
帮我新增一道番茄鸡蛋面的菜谱，2 人份，做法简单一点。
```

预期：

- 生成菜谱草稿审批。
- 确认前菜谱库不应出现新菜谱。
- 确认后菜谱库出现该菜谱，并同步生成可用食物资料。

输入：

```text
把番茄鸡蛋面改成 3 人份，步骤里提醒先炒番茄出汁。
```

预期：

- AI 定位真实菜谱。
- 生成 `recipe_operation.v1` 更新审批。
- 面板显示当前菜谱与修改字段。
- 确认后旧菜谱被更新，不新增重复菜谱。

输入：

```text
删除番茄鸡蛋面这道菜谱。
```

预期：

- 生成删除审批，显示关联影响。
- 确认后菜谱被删除，关联的家常菜食物同步处理。

### 4.5 meal_plan

输入：

```text
把明天晚餐安排成番茄炒蛋。
```

预期：

- AI 使用真实食物 ID 创建计划草稿。
- 确认后明天晚餐出现番茄炒蛋。

输入：

```text
把明天晚餐改成盒装牛奶。
```

预期：

- 如果明天晚餐有多个计划，AI 要求消歧。
- 若目标明确，生成更新审批。
- 确认后原计划被修改，不新增重复计划。

输入：

```text
把明天晚餐标记为已完成。
```

预期：

- 生成 `set_status` 审批。
- 确认后计划状态变为完成。

### 4.6 shopping_list

输入：

```text
把鸡蛋加入购物清单，数量 2 盒，原因是早餐快用完了。
```

预期：

- 生成购物清单创建审批。
- 确认后清单出现鸡蛋 2 盒。

输入：

```text
把购物清单里的鸡蛋标记为已买。
```

预期：

- AI 定位真实购物项。
- 生成 `set_done` 审批。
- 确认后购物项变成已完成。

输入：

```text
删除购物清单里的鸡蛋。
```

预期：

- 生成删除审批。
- 确认后该购物项不再显示在待采购列表。

### 4.7 meal_log

输入：

```text
记录今晚吃了番茄炒蛋，1 份，心情不错。
```

预期：

- AI 匹配真实食物“番茄炒蛋”。
- 生成餐食记录审批。
- 确认后餐食记录出现新条目。

输入：

```text
给刚才那顿番茄炒蛋打 4.5 分。
```

预期：

- AI 定位真实餐食记录和食物项。
- 生成评分审批。
- 确认后该食物项评分更新。

### 4.8 recipe_cook

输入：

```text
开始做番茄炒蛋，按 2 人份，做完后记录到今晚晚餐。
```

预期：

- AI 读取菜谱和库存。
- 生成做菜审批，显示扣减批次和缺料。
- 如果库存不足，应提示不能直接完成并建议采购或补库存。
- 如果库存足够，确认后：
  - 库存扣减。
  - 生成做菜记录。
  - 按请求生成餐食记录。
  - 如果关联了计划项，计划状态变为完成。

## 5. 安全和异常测试

### 5.1 拒绝审批

任意创建一个菜谱或计划审批，在审批面板点击拒绝。

预期：

- 审批状态变为 rejected。
- 草稿状态变为 rejected。
- 正式菜谱、计划、购物项等业务表不变化。

### 5.2 并发冲突

步骤：

1. 让 AI 生成“修改明天晚餐计划”的审批，但先不要确认。
2. 在页面或数据库中手动修改同一计划。
3. 回到 AI 审批点击确认。

预期：

- 审批执行失败。
- 系统生成重试审批。
- 面板显示当前值和恢复提示。
- 原业务对象不被旧草稿覆盖。

### 5.3 模糊目标

准备两个名称相近的计划或购物项，然后输入：

```text
把鸡蛋标记为已买。
```

预期：

- 如果存在多个候选，AI 不应直接选择。
- AI 应要求用户确认具体目标。

### 5.4 跨家庭目标

该项通常需要接口或测试数据辅助。尝试让 AI 修改不属于当前家庭的食物、菜谱、计划或购物项。

预期：

- Tool normalize 或正式提交阶段拒绝。
- 错误应表达“不属于当前家庭”或“不属于当前用户”。
- 不产生业务副作用。

## 6. 诊断弹窗测试

桌面端点击 AI 状态胶囊。

预期：

- 打开 AI 质量诊断弹窗。
- 可看到最近 run、Skill 分布、审批统计、澄清原因等。
- 默认会话侧栏不再常驻诊断卡片。

## 7. 记录测试结果

建议手动测试时记录：

- 输入 prompt。
- AI 路由 Skill。
- 生成的审批类型。
- 是否出现确认前正式写入。
- 确认或拒绝后的业务结果。
- 是否出现结果卡。
- 异常情况下是否有重试审批或澄清提示。
